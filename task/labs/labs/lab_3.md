# Лабораторная 3
# Цель

Получение практических навыков по обеспечению безопасности веб-приложения на всех этапах жизненного цикла.

# Задачи
В соответствии с требованиями к проекту (см. `/requirements.md`):
1. Разработать интерфейсы управления, согласно требованиям (см. `/requirements.md`).
2. Обеспечить безопасность интерфейсов взаимодействия с системой.
3. Обеспечить безопасность среды эксплуатации системы.
4. Настроить процесс непрерывной интеграции (CI/CD)
5. Разработать оставшуюся документацию на систему.

# Требования к результату

>

## 1 Требования к интерфейсу

- [ ] Графический интерфейс
- [ ] Адаптация под мобильные устройства
- При изменении размера окна браузера (эмуляция мобильных устройств) элементы интерфейса не должны выходить за границы экрана.
- Должен быть использован мета-тег viewport.
- Навигация должна быть удобной для использования на сенсорных экранах (минимальный размер интерактивных элементов не менее 44px).

## 2 Требования к безопасности

**Политики SOP, CSP, CORS:**
- [ ] SOP: Браузер должен блокировать скрипты, пытающиеся взаимодействовать с источниками, отличными от origin приложения.
- [ ] CSP: В ответах сервера должен присутствовать заголовок Content-Security-Policy. Должны быть строго заданы директивы default-src, script-src, style-src.
- [ ] CORS: Сервер должен возвращать заголовок Access-Control-Allow-Origin только для заранее определенного списка доверенных доменов. Для неавторизованных источников доступ должен быть запрещен (код 403).

**Защита форм и API:**
- [ ] CSRF: в каждой HTML-форме должен присутствовать уникальный токен, который валидируется на сервере. Токен не должен передаваться через GET-параметры.
- [ ] API Keys / Токены: Доступ к методам API должен быть ограничен. Запросы без валидного токена (Bearer или API Key) должны отклоняться с кодом **401**.
- [ ] Rate Limiting: Должна быть настроена защита от DoS-атак. При превышении лимита запросов (10 запросов в минуту) с одного IP-адреса или токена, API должен возвращать код **429**.
- [ ] Журналирование: Каждый входящий HTTP-запрос должнен логироваться. В логах должен присутствовать уникальный идентификатор Request ID формата **UUIDv7**. Этот ID должен пробрасываться во все сервисы и возвращаться в заголовке X-Request-Id клиенту.

## 3 Обеспечение безопасности среды эксплуатации
### Настройка безопасности сервера эксплуатации ПО (hardering)
Сервер эксплуатации, где развернут дистрибутив ПО, должен удовлетворять следующим требованиям:
- [ ] Настроена безопасность удаленного подключения:
	- Запрещен вход под пользователем root по SSH (используется отдельный пользователь).
	- Настроена политика сложных паролей для пользователя;
	- Включена защита от перебора паролей;
	- Настроено подключение по сертификатам;
	- Порт перенесен на нестандартный.
- [ ] Настроен межсетевой экран (например, ufw), разрешающий только порты удаленного подключения и необходимые приложению.
- [ ] Отсутствуют процессы, запущенные с избыточными привилегиями (приложение не запускается от root, используют readonly ФС по умолчанию).

### Доступ к приложению через веб-сервер (Nginx) и защитой проколом TLS
Настройка веб-сервера Nginx должна удовлетворять следующим требованиям:
- [ ] Nginx должен выступать в роли шлюза (gateway) и работать в режиме обратного прокси (reverse-proxy), направляя весь трафик к запущенному веб-приложению.
- [ ] На веб-сервере должен быть настроен TLS-сертификат при помощи утилиты Certbot.
- [ ] Заголовок HSTS: доступ к приложению должен быть возможен строго по протоколу HTTPS.
### Мониторинг ресурсов системы с помощью инструментов Prometheus и Grafana 
Должна быть настроена система мониторинга состояния разработанной системы на основе системы Prometheus и веб-клиента Grafana:
- [ ] На сервер эксплуатации должен быть развернут сервис сбора метрик Prometheus Node exporter (https://github.com/prometheus/node_exporter). 
- [ ] Сервис Node exporter должен собирать метрики использования ЦПУ (CPU) и ОЗУ (RAM) сервера.
- [ ] На сервере эксплуатации должен быть развернут веб-килент анализа и визуализации Grafana (https://github.com/grafana/grafana): 
- [ ] веб-клиент должен быть интегрирован с системой мониторинга Prometheus;
- [ ] должна быть настроена панель мониторинга состояния системы (dashboard), отображающий минимум 3 графика:
	1. Использование CPU и RAM сервера.
	2. Количество HTTP запросов в секунду (requests-per-second) с разбивкой по кодам ответов (2xx, 4xx, 429).
	3. Количество зафиксированных ошибок Rate Limit (HTTP 429).

## 4 Настройка процесса непрерывной интеграции (CI/CD)

### Git-репозиторий с системой версионирования дистрибутивов
- [ ] Исходный код ПО должен храниться в Git-репозитории.
- [ ] Версии дистрибутивов должны быть оформлены с помощью Git-тегов со структурой vX.Y.Z. Необходимо описывать изменения мажорной, минорной и патч-версий.
### Настройка безопасного хранения переменных окружения 
- [ ] Пароли, токены и другие авторизационные данные должны хранится в CI/CD Variables и автоматически подставляться по время запуска процесса автоматизации сборки (pipeline).
### Настройка статического анализатора кода SAST (Static Application Security Testing):
- [ ] Должен запускаться этап статического анализа кода при помощи **SonarQube**.
- [ ] Отчет анализатора должен показывать отсутствие критических уязвимостей.
### Настройка сканирования безопасности Docker-контейнеров (Container Scanning)
- [ ] Должен запускаться этап сканирования образа утилитой **Aqua Trivy**.
- [ ] Отчет о найденных уязвимостях в слоях образа и зависимостях приложения.
### Автоматическая сборка дистрибутивов ПО системы
- [ ] При отправке кода (push) в ветку main/master или создании тега процесс автоматизации сборки дистрибутива (pipeline) должен содержать следующие этапы:
	1. Установка зависимостей.
	2. Запуск статического анализатора правил написания исходного кода (linters) 
	3. Запуск статического анализа на безопасность исходного кода (SAST).
	4. Сборка дистрибутива: 
		- артефакт дистрибутива должен находиться в релизах (Releases);
		- Docker-образ дистрибутива должен находиться в реестре дистрибутивов Gitlab (Gitlab Container Registry).

Развертывание на сервере эксплуатации (CD) опционально.
