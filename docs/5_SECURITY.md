# Описание мер обеспечения безопасности системы

## Обеспечение безопасности интерфейсов взаимодействия с системой

### 1. Аутентификация и авторизация

**1.1 JWT-токены**
- Токены доступа имеют срок жизни 24 часа
- Подпись токенов осуществляется с использованием HMAC-SHA256
- Токен содержит идентификатор пользователя, роль и ID сессии
- При каждом запросе проверяется валидность подписи и срок действия

**1.2 API-ключи**
- Формат ключа: `hg_v1_{256-битный хеш}`
- В базе данных хранится только SHA-256 хеш ключа (сам ключ не хранится)
- API-ключи не могут создавать или удалять другие API-ключи
- Поддержка опционального срока действия ключа
- Возможность отзыва ключа без удаления из БД

**1.3 Двухфакторная аутентификация**
- Реализована через TOTP (RFC 6238)
- Секрет генерируется при регистрации (20 байт, Base32)
- QR-код для настройки приложения-аутентификатора
- Проверка кода на серверной стороне

**1.4 OAuth-авторизация**
- Поддержка Google и GitHub
- Защита от CSRF через параметр state
- Привязка нескольких OAuth-провайдеров к одному пользователю
- Унифицированные сообщения об ошибках для предотвращения перебора

### 2. Защита от атак

**2.1 Rate Limiting**
- Лимиты настроены для каждого эндпоинта отдельно
- Заголовки ответа: X-RateLimit-Limit, X-RateLimit-Remaining, Retry-After
- Код ответа при превышении: 429 Too Many Requests

| Эндпоинт | Лимит (запросов/мин) | Burst |
|----------|---------------------|-------|
| /api/login | 5 | 5 |
| /api/users | 10 | 10 |
| /api/check | 100 | 50 |
| /api/behavior/ | 300 | 100 |
| /api/files/upload | 10 | 5 |
| /api/sites | 30 | 15 |
| /api/admin/ | 20 | 10 |

**2.2 CSRF-защита**
- Генерация случайного токена (32 байта, Base64)
- Токен хранится в HttpOnly cookie
- Для мутирующих запросов (POST/PUT/DELETE) требуется заголовок X-CSRF-Token
- Исключения: /api/login, /api/users, /api/check, /api/behavior/, /api/auth/

**2.3 CORS-политика**
- Разрешённые источники задаются через переменную окружения CORS_ALLOWED_ORIGINS
- Поддержка нескольких доменов через запятую
- Для авторизованных запросов обязательно указание конкретного origin
- Заголовок Access-Control-Allow-Credentials установлен в true

**2.4 Content Security Policy**
- Production: script-src 'self', style-src 'self'
- Development: дополнительные директивы для Hot Module Replacement
- Запрет загрузки ресурсов с неподтверждённых источников

**2.5 Защита от брутфорса**
- Унифицированные сообщения об ошибках при логине: "invalid email or password or verification code"
- Невозможно определить по ответу, существует ли email или правильный ли пароль
- Rate limit 5 попыток в минуту

### 3. Безопасность данных

**3.1 Хранение паролей**
- Использование bcrypt с cost = 10
- Невозможность восстановления исходного пароля из хеша

**3.2 Защита чувствительных данных**
- Поля PasswordHash, TOTPSecret, OAuthID не возвращаются в API-ответах
- Пароли не логируются
- Логи не содержат чувствительной информации

**3.3 Безопасность cookie**
| Cookie | HttpOnly | Secure | SameSite | MaxAge |
|--------|----------|--------|----------|--------|
| csrf_token | Да | Да | Strict | 24 часа |
| hg_session | Да | Да | Lax | 30 минут |

### 4. Сквозное логирование

**4.1 Request ID**
- Каждый запрос получает уникальный X-Request-ID (UUIDv7)
- ID присутствует во всех логах, связанных с запросом
- Возвращается клиенту в заголовке ответа

**4.2 Формат логов**
- Время запроса
- Request ID
- HTTP метод и путь
- Время выполнения
- Метод аутентификации (jwt/api_key/none)

## Обеспечение безопасности среды эксплуатации ПО

### 1. Контейнеризация

**1.1 Docker-образы**
- Мультистейдж сборка для минимизации размера
- Запуск от непривилегированного пользователя (не root)
- Использование официальных базовых образов (alpine)

**1.2 Docker Compose**
- Изоляция сервисов в отдельной сети (backend_network)
- Отсутствие публикации портов для внутренних сервисов (postgres, minio, keycloak)
- Использование именованных томов для персистентных данных

### 2. Сетевая безопасность

**2.1 Reverse proxy (Nginx)**
- Терминация TLS
- Передача реального IP клиента через X-Real-IP и X-Forwarded-For
- Ограничение максимального размера тела запроса (25 MB)
- Заголовки безопасности:
  - Strict-Transport-Security: max-age=31536000; includeSubDomains; preload
  - X-Frame-Options: SAMEORIGIN
  - X-Content-Type-Options: nosniff

**2.2 TLS/SSL**
- Использование Let's Encrypt для получения сертификатов
- Автоматическое обновление сертификатов
- Поддержка только TLSv1.2 и TLSv1.3

### 3. Мониторинг безопасности

**3.1 Сканирование уязвимостей (CI/CD)**
- Trivy сканирование Docker-образов на наличие известных уязвимостей
- Критический и высокий уровень уязвимостей блокируют сборку

**3.2 Статический анализ кода (SAST)**
- Semgrep для поиска уязвимостей в коде
- SonarQube для анализа качества и безопасности

**3.3 Метрики и алерты (Prometheus/Grafana)**
- Активные сессии
- HTTP ошибки (4xx, 5xx, 429)
- Rate limit превышения

### 4. Управление секретами

**4.1 Переменные окружения**
- Все секреты передаются через переменные окружения (12-factor app)
- Примеры конфигурации в .env.example (без реальных секретов)
- В production секреты хранятся в CI/CD secrets, не в репозитории

**4.2 Обязательные секреты**
- JWT_SECRET — ключ подписи токенов
- POSTGRES_PASSWORD — пароль БД
- MINIO_ROOT_PASSWORD — пароль S3-хранилища
- OAuth Client Secrets для Google, GitHub, Keycloak

### 5. База данных

**5.1 PostgreSQL**
- Пароль пользователя задаётся через переменную окружения
- Отсутствие публикации порта 5432 в production
- Регулярное резервное копирование (на уровне инфраструктуры)

**5.2 Миграции**
- Миграции применяются при запуске контейнера
- Поддержка отката через down-миграции
- Все изменения схемы — только через миграции

### 6. Сборка и деплой (CI/CD)

**6.1 GitHub Actions**
- Автоматическая сборка при создании тега
- Проверка кода линтерами перед сборкой
- Подпись коммитов и тегов

**6.2 Релизы**
- Семантическое версионирование (vMAJOR.MINOR.PATCH)
- Релизный архив содержит docker-compose.yml и миграции
- Docker-образы публикуются в GitHub Container Registry (ghcr.io)

### 7. Рекомендации по эксплуатации

**7.1 Для администратора системы**
- Регулярно обновлять версии HumanGuard
- Использовать длинные случайные пароли для JWT_SECRET
- Ограничить доступ к серверу по SSH (только ключи)
- Настроить автоматическое обновление сертификатов Let's Encrypt
- Включить аудит логов

**7.2 Для владельца сайта**
- Использовать сложные пароли
- Включить 2FA в профиле
- Регулярно проверять блэклист и подозрительные сессии
- Настраивать пороги risk score под свой трафик
