# Техническое задание

## Термины и определения

| Термин | Определение |
|--------|-------------|
| HumanGuard | Система защиты веб-сайтов от ботов и автоматизированных атак |
| Владелец сайта | Пользователь, зарегистрировавший сайт в системе |
| Risk score | Числовой показатель (0-100), отражающий вероятность бот-активности |
| Блэклист | Список IP-адресов, заблокированных для конкретного сайта |
| Visitor session | Сессия неавторизованного посетителя для анализа поведения |

## 1 Функциональные требования

### 1.1 Требования к управлению пользователями

#### Функциональное требование 1.1.1: Регистрация пользователя
- **Описание:** пользователь создаёт учётную запись по email и паролю (мин. 8 символов)
- **Триггер:** POST /api/users
- **Результат:** создан пользователь с ролью "user", сгенерирован TOTP-секрет для 2FA

#### Функциональное требование 1.1.2: Автосоздание администратора
- **Описание:** при первом запуске создаётся администратор из переменных окружения
- **Триггер:** запуск бекенда при отсутствии пользователя с DEFAULT_ADMIN_EMAIL
- **Результат:** создан пользователь с ролью "admin"

#### Функциональное требование 1.1.3: Просмотр списка пользователей
- **Описание:** администратор просматривает всех зарегистрированных пользователей
- **Триггер:** GET /api/users (только роль admin)
- **Результат:** список пользователей без паролей и TOTP-секретов

#### Функциональное требование 1.1.4: Редактирование пользователя
- **Описание:** администратор изменяет имя и роль пользователя
- **Триггер:** PUT /api/users/{id}
- **Результат:** обновлённые данные пользователя

#### Функциональное требование 1.1.5: Удаление пользователя
- **Описание:** администратор удаляет пользователя и все связанные данные
- **Триггер:** DELETE /api/users/{id}
- **Результат:** пользователь и его OAuth-связи удалены

#### Функциональное требование 1.1.6: Управление API-ключами
- **Описание:** пользователь создаёт, просматривает и отзывает API-ключи
- **Триггер:** POST/GET/DELETE /api/keys
- **Результат:** сгенерирован ключ формата hg_v1_{хеш} с опциональным сроком действия

#### Функциональное требование 1.1.7: Деактивация сессий
- **Описание:** администратор принудительно завершает сессию пользователя
- **Триггер:** DELETE /api/admin/users/sessions/{session_id}
- **Результат:** сессия удалена, пользователь вынужден перелогиниться

### 1.2 Требования к аутентификации и авторизации

#### Функциональное требование 1.2.1: Аутентификация по email и паролю
- **Описание:** пользователь входит в систему с поддержкой 2FA
- **Триггер:** POST /api/login
- **Результат:** выдан JWT-токен (TTL 24 часа), создана сессия с реальным IP

#### Функциональное требование 1.2.2: Унифицированные ошибки логина
- **Описание:** при любом ошибочном вводе возвращается одинаковое сообщение
- **Триггер:** неверный email, пароль или TOTP-код
- **Результат:** `{"error":"invalid email or password or verification code"}`

#### Функциональное требование 1.2.3: OAuth-авторизация через Google
- **Описание:** пользователь входит через Google аккаунт
- **Триггер:** GET /api/auth/google/login
- **Результат:** создан/найден пользователь, привязан OAuth, выдан JWT-токен

#### Функциональное требование 1.2.4: OAuth-авторизация через GitHub
- **Описание:** пользователь входит через GitHub аккаунт
- **Триггер:** GET /api/auth/github/login
- **Результат:** создан/найден пользователь, привязан OAuth, выдан JWT-токен

#### Функциональное требование 1.2.5: Привязка нескольких OAuth-провайдеров
- **Описание:** при входе через нового провайдера с тем же email происходит привязка
- **Триггер:** OAuth-колбэк с email уже зарегистрированного пользователя
- **Результат:** создана запись в таблице user_oauth

#### Функциональное требование 1.2.6: Смена пароля
- **Описание:** пользователь меняет свой пароль (требуется старый пароль)
- **Триггер:** POST /api/users/{id}/password
- **Результат:** пароль обновлён

#### Функциональное требование 1.2.7: Смена аватара
- **Описание:** пользователь устанавливает URL аватара
- **Триггер:** POST /api/users/{id}/avatar
- **Результат:** avatar_url обновлён

#### Функциональное требование 1.2.8: Выход из системы
- **Описание:** пользователь завершает сессию
- **Триггер:** POST /api/logout
- **Результат:** сессия удалена из памяти

#### Функциональное требование 1.2.9: Проверка текущего пользователя
- **Описание:** получение данных авторизованного пользователя
- **Триггер:** GET /api/me
- **Результат:** данные пользователя (без пароля и TOTP)

### 1.3 Требования к управлению сайтами

#### Функциональное требование 1.3.1: Создание сайта
- **Описание:** владелец регистрирует сайт для защиты
- **Триггер:** POST /api/sites
- **Результат:** создан сайт со статусом "verifying"

#### Функциональное требование 1.3.2: Просмотр списка сайтов
- **Описание:** владелец видит все свои сайты
- **Триггер:** GET /api/sites
- **Результат:** список сайтов пользователя

#### Функциональное требование 1.3.3: Активация/приостановка сайта
- **Описание:** владелец включает или выключает защиту для сайта
- **Триггер:** POST /api/sites/{id}/activate, POST /api/sites/{id}/suspend
- **Результат:** статус сайта изменён

#### Функциональное требование 1.3.4: Удаление сайта
- **Описание:** владелец удаляет сайт из системы
- **Триггер:** DELETE /api/sites/{id}
- **Результат:** сайт и связанные данные удалены

#### Функциональное требование 1.3.5: Просмотр настроек сайта
- **Описание:** владелец получает текущую конфигурацию защиты
- **Триггер:** GET /api/sites/{id}/settings
- **Результат:** JSON с настройками collector, analyzer, reaction

#### Функциональное требование 1.3.6: Обновление настроек сайта
- **Описание:** владелец изменяет параметры защиты
- **Триггер:** PUT /api/sites/{id}/settings
- **Результат:** настройки сохранены в JSONB поле settings

### 1.4 Требования к блэклисту

#### Функциональное требование 1.4.1: Просмотр блэклиста
- **Описание:** владелец видит заблокированные IP для своего сайта
- **Триггер:** GET /api/sites/{id}/blacklist
- **Результат:** список IP, причин и дат блокировки

#### Функциональное требование 1.4.2: Добавление IP в блэклист
- **Описание:** владелец вручную блокирует IP-адрес
- **Триггер:** POST /api/sites/{id}/blacklist
- **Результат:** IP добавлен, запросы с него отклоняются

#### Функциональное требование 1.4.3: Удаление IP из блэклиста
- **Описание:** владелец разблокирует IP-адрес
- **Триггер:** DELETE /api/sites/{id}/blacklist/{ip}
- **Результат:** IP удалён из блэклиста

### 1.5 Требования к детекции ботов

#### Функциональное требование 1.5.1: Проверка запроса к сайту
- **Описание:** система оценивает запрос посетителя и возвращает действие
- **Триггер:** POST /api/check с заголовком X-Site-ID
- **Результат:** создана/обновлена сессия, возвращено action (allow/captcha/block)

#### Функциональное требование 1.5.2: Сбор метрик поведения
- **Описание:** клиентский трекер отправляет данные о поведении посетителя
- **Триггер:** POST /api/behavior/{id}
- **Результат:** метрики сохранены в сессию

#### Функциональное требование 1.5.3: Обнаружение headless-браузеров
- **Описание:** детектор анализирует User-Agent на признаки автоматизации
- **Триггер:** вызов Prefilter в детекторе
- **Результат:** повышение risk score при обнаружении headless

#### Функциональное требование 1.5.4: Rate limiting по IP
- **Описание:** ограничение частоты запросов с одного IP
- **Триггер:** превышение лимита в middleware
- **Результат:** HTTP 429 Too Many Requests

### 1.6 Требования к работе с файлами

#### Функциональное требование 1.6.1: Загрузка файла
- **Описание:** пользователь загружает файл в систему
- **Триггер:** POST /api/files/upload (multipart/form-data)
- **Результат:** файл сохранён в S3 или локальное хранилище

#### Функциональное требование 1.6.2: Скачивание файла
- **Описание:** пользователь скачивает свой файл
- **Триггер:** GET /api/files/{id}
- **Результат:** файл передан в ответе

#### Функциональное требование 1.6.3: Удаление файла
- **Описание:** пользователь удаляет свой файл
- **Триггер:** DELETE /api/files/{id}
- **Результат:** файл удалён из хранилища и БД

#### Функциональное требование 1.6.4: Создание ссылки для шаринга
- **Описание:** пользователь создаёт временную публичную ссылку на файл
- **Триггер:** POST /api/files/share
- **Результат:** сгенерирован уникальный токен

#### Функциональное требование 1.6.5: Загрузка по шаринг-ссылке
- **Описание:** любой пользователь скачивает файл по токену
- **Триггер:** GET /api/files/share/{token}
- **Результат:** файл передан (если ссылка не истекла)

## 2 Требования к интерфейсам

### 2.1 Требования к графическому интерфейсу администратора

#### Функциональное требование 2.1.1: Панель управления сайтами
- Отображение списка сайтов с кнопками Activate/Suspend/Delete
- Форма создания нового сайта (название, домен, origin server, настройки JSON)
- Визуальное отображение статуса сайта (active/suspended/verifying)

#### Функциональное требование 2.1.2: Управление чёрным списком
- Отображение заблокированных IP для каждого сайта
- Форма добавления IP с возможностью указать причину
- Кнопка удаления IP из блэклиста

#### Функциональное требование 2.1.3: Управление пользователями
- Таблица всех пользователей с email, ролью, датой регистрации
- Редактирование роли (user/admin)
- Удаление пользователя

#### Функциональное требование 2.1.4: Управление API-ключами
- Список ключей с префиксом и датой создания
- Форма создания ключа (название, срок действия)
- Отображение ключа (только один раз)
- Отзыв ключа

#### Функциональное требование 2.1.5: Управление профилем
- Отображение информации о пользователе
- Смена пароля (требуется старый пароль)
- Смена аватара через URL
- Переключение темы (стандартная, cyan, тёмная)

### 2.2 Требования к внешним интерфейсам HTTP API

#### Функциональное требование 2.2.1: Авторизация
- JWT токен в заголовке `Authorization: Bearer <token>`
- API ключ в заголовке `X-API-Key: hg_v1_...`

#### Функциональное требование 2.2.2: Формат ответов
- Успех: HTTP 200/201/204, Content-Type: application/json
- Ошибки: HTTP 4xx/5xx с JSON-телом по RFC 9457

#### Функциональное требование 2.2.3: Rate limiting
- Ограничение на каждый эндпоинт
- Заголовки: X-RateLimit-Limit, X-RateLimit-Remaining, Retry-After
- При превышении: HTTP 429 Too Many Requests

#### Функциональное требование 2.2.4: Request ID
- Каждый ответ содержит X-Request-ID (UUIDv7)
- Логи привязаны к этому ID

#### Функциональное требование 2.2.5: CORS
- Заголовки для доступа с фронтенда
- Allow-Origin из CORS_ALLOWED_ORIGINS

#### Функциональное требование 2.2.6: CSRF защита
- Для POST/PUT/DELETE требуется X-CSRF-Token
- Исключения: /api/login, /api/users, /api/auth/

#### Функциональное требование 2.2.7: Swagger документация
- OpenAPI v3 в docs/api/swagger.yaml

### 2.3 Требования к графическому интерфейсу пользователя

#### Функциональное требование 2.3.1: Страница входа
- Форма с email, паролем
- Вход через Google и GitHub
- Поддержка TOTP-кода

#### Функциональное требование 2.3.2: Страница регистрации
- Форма с email, паролем (мин. 8 символов), именем
- Отображение QR-кода для 2FA и секрета

#### Функциональное требование 2.3.3: Дашборд
- Переключение темы
- Панель отладки поведения (для админа)

#### Функциональное требование 2.3.4: Управление файлами
- Загрузка с прогрессом (WebSocket)
- Список файлов с кнопками "Скачать" и "Share"
- Отображение размера и даты

#### Функциональное требование 2.3.5: Адаптивный дизайн
- Корректное отображение на мобильных устройствах (от 320px)
- Меню-гамбургер на маленьких экранах

## 3 Нефункциональные требования

### 3.1 Требования к обработке больших данных

#### Функциональное требование 3.1.1: Загрузка больших файлов
- Поддержка файлов до 5 ГБ
- Прогресс через WebSocket
- Асинхронная обработка

#### Функциональное требование 3.1.2: Масштабирование сессий
- Visitor сессии в памяти с TTL 30 минут
- Автоочистка каждые 5 минут

#### Функциональное требование 3.1.3: Лимиты по умолчанию
- Максимальный размер файла: 5 ГБ
- JWT TTL: 24 часа
- User session TTL: 24 часа
- Visitor session TTL: 30 минут

### 3.2 Требования к безопасности

#### Функциональное требование 3.2.1: Защита паролей
- Хранение bcrypt-хешей (cost=10)
- Пароль не возвращается в API

#### Функциональное требование 3.2.2: Защита от брутфорса
- Rate limit 5 запросов/мин на /api/login
- Унифицированные сообщения об ошибках

#### Функциональное требование 3.2.3: Защита от утечки информации
- В API скрыты PasswordHash, TOTPSecret, OAuthID
- Логи без чувствительных данных

#### Функциональное требование 3.2.4: Безопасность cookie
- HttpOnly: true (csrf_token, hg_session)
- Secure: true
- SameSite: Strict/Lax

#### Функциональное требование 3.2.5: Защита от CSRF
- X-CSRF-Token для мутирующих запросов
- Токен 32 байта, в HttpOnly cookie

#### Функциональное требование 3.2.6: Content Security Policy
- Production: script-src 'self', style-src 'self'
- Development: добавлены unsafe-inline/eval/localhost

#### Функциональное требование 3.2.7: Защита API-ключей
- Префикс hg_v1_
- Хранится SHA-256 хеш
- Ключи не управляют другими ключами

### 3.3 Требования к инфраструктуре

#### Функциональное требование 3.3.1: Контейнеризация
- Docker-образы с мультистейдж сборкой
- Docker Compose для разработки и продакшена

#### Функциональное требование 3.3.2: База данных
- PostgreSQL 15
- Миграции через .sql файлы
- Индексы на FK и часто запрашиваемые поля

#### Функциональное требование 3.3.3: Объектное хранилище
- Поддержка MinIO и локального хранилища
- Переключение через STORAGE_TYPE

#### Функциональное требование 3.3.4: Обратный прокси
- Nginx с терминированием TLS
- Передача X-Real-IP, X-Forwarded-For

#### Функциональное требование 3.3.5: Мониторинг
- Prometheus метрики на /metrics
- Метрики: HTTP запросы, активные сессии, risk score
- Grafana дашборд

#### Функциональное требование 3.3.6: CI/CD
- GitHub Actions: линтеры, SAST, Trivy
- Сборка Docker-образов при создании тега
- GitHub Release с архивом

#### Функциональное требование 3.3.7: Семантическое версионирование
- Теги vMAJOR.MINOR.PATCH
- Версия в интерфейсе

#### Функциональное требование 3.3.8: Конфигурация
- 12-factor config через переменные окружения
- Примеры в .env.example

#### Функциональное требование 3.3.9: Сквозное логирование
- X-Request-ID (UUIDv7) на каждый запрос
- Логи: requestID, метод, путь, время, auth-метод
- Вывод в stdout
